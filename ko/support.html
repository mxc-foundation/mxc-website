LPWAN Server

MXC Foundation's LPWAN Server is a LoRaWAN application-server. It can be used as a Web App to manage devices part of a LoRaWAN network, as well as applications. Individuals and Organisations can easily manage devices and applications to trade data without API knowledge.

See also the complete list of LPWAN Server features.

Requirements

To use MXC Foundation's LPWAN Server you are required to login with a valid email.

Get Started with your LPWAN Server

Welcome to MXProtocol. In only a few steps you are going to be part of a state of the art network for machine-to-machine communication. Remember to have in hand your Gateway MAC Address and your device EUI number.

  1. To be a part of the LPWAN network you only need a valid email. So start by clicking Register and submitting your email.
  2. You are going to receive a confirmation email with a link to proceed with your registration. If that email does not arrive, please check your Spam folder or consider re-submitting your registration request. Click or paste the URL into your browser.
  3. Complete the registration form and pick a name for your Organisation. Learn more about Organisations here. Please note that your Organisation name is to be added with no spaces between characters.
  4. Go to Organisations and click the name of your Organisation. Make sure that ‘Organisation can have gateways’ option is enabled. Click Update.
  5. Now click All Users to make one of them Admin of your Organisation. Click the username of your choice and enable Permissions ‘Is active’ and ‘Is Global Admin’. Click Update.
  6. To set up your LPWAN Server Network, click Network-Servers and Add your first one. Add a Name and a Server Port. Please note that Name must not user spaces between characters. The default Port is 'localhost:8000'.
    1. On Gateway Discovery Tab, please Check the Enable gateway discovery checkbox.
  7. Now you are ready to set up a gateway. Click Gateways-profiles and then Create.
    1. Enter the name you like for your gateway profile.
    2. Enter '0, 1, 2, 3, 4, 5, 6, 7' as channels.
    3. Choose loraserver as your Network-server.
  8. Now you can add you first Service-profile. Each profile can be used to configure worldwide gateways, with a specific service. Click Service-profiles and Create.
    1. Enter a name. Please note that the name is to be added with no spaces between characters.
    2. Select loraserver as Network Server and enable Gateway Metadata and Network Geolocation by selecting the checkboxes.
    3. Set Device status request frequency to 0. Set Minimum and maximum allowed data rate to 10 and click Create.
  9. Next step is to add Device-profile. First enter Device-profile name. Please note that the name is to be added with no spaces between characters.
    1. Select loraserver as Network-server.
    2. Select 1.0.0 as LoRaWAN MAC version. Click on JOIN (OTAA/ABP) tab and then select the following ‘Device Support OTAA’. Click Update.
  10. Next step is adding your Gateway. Click Gateways and Create. First, enter a name for your Gateway. Please note that the name is to be added with no spaces between characters.
    1. Add description. Enter MAC address as Gateway ID and add Loraserver as your Network-server.
    2. Select your previously created Device-profile and enable Gateway discovery by clicking the checkbox. Click Create.
  11. Add a new Application. Applications are instances that contain your end devices. Select the Application menu and click Create.
    1. Enter the Application name. Please note that the name is to be added with no spaces between characters.
    2. Enter the Gateway description and select the previously created Service-profile. Click Create.
  12. Finally, add your device by reaching back to the previous menu and clicking on the Application you just created. Go to the Device tab and click Create.
    1. Enter the device name. Please note that the name is to be added with no spaces between characters.
    2. Enter the description and device EUI number. Select the Device-profile you previously created. Click Create to save your device.
    3. Go to the KEYS (OTAA) tab, enter the provided Application Key and click on the SET-DEVICE-KEYS button.

Features

User management

An unlimited number of authenticated users can be created on LPWAN Server. Users can have different levels of authorization within the Organisation: regular, admin or super-admin, for supernode owners.

Admin User

An admin user is authorized to perform any action within LPWAN Server. It can for example manage gateways, users, create Organisations, applications and nodes.

User

A regular user has reading only permition by default.

Network-server management

Network-servers

LPWAN Server is able to connect to one or multiple network-server instances. Admin users are able to add new network-servers. Once a network-server is assigned to a service-profile or device-profile, the network-server can’t be removed before deleting these entities, it will return an error.

Routing-profile

When creating a new network-server, LPWAN App Server will create a routing-profile on the given network-server, containing the hostname:ip of the LPWAN App Server installation. In case your LPWAN App Server installation is not reachable on localhost, make sure this hostname:ip is configured correctly in your configuration. This routing-profile is updated on network-server updates and deleted on network-server deletes.

TSL Certificates

Depending the configuration of LPWAN Server and LPWAN App Server, you must enter the CA and client certificates in order to let LPWAN App Server connect to LPWAN Server and in order to let LPWAN Server connect to LPWAN App Server (see Routing-profile above). For security reasons, the TLS key content is not displayed when editing an existing network-server. Re-submitting does not clear the stored TLS key when left blank! The TLS key content will only be cleared internally when submitting a form with an empty TLS certificate value.

Gateway-profiles

Once a network-server has been created, it is possible to provision one or more gateway-profiles on this network-server (available under the Gateway-profiles tab). When adding a gateway, it is then possible to select one of these gateway-profiles to make sure the gateway configuration is in sync with the channels used by the network.

Important: changing the channel-plan of the gateway does not change the channels used by the devices. For a correct setup, the gateway-profile(s) must at least support the channels used by your devices. The channels that must be used by your devices must be configured in your LPWAN Server configuration.

Enabled channels

Enter the list of default channels that you would like to use for this gateway-profile. Note that these are the channels that are specified by the LoRaWAN Regional Parameters.

Extra channels

If allowed by your LoRaWAN region, you can use Add extra channel to configure additional channels that are not defined by the LoRaWAN Regional Parameters.

Hardware limitations

This feature is limited to 8-channel gateways (currently) and assumes that channels can be distributed over two radios. When defining a channel-plan, please keep in mind that the channels fit within the bandwidth of two radios.

The bandwidth of each radio depends on the bandwidth of the assigned channels:

  • 500kHz channel = 1.1MHz radio bandwidth
  • 250kHz channel = 1Mhz radio bandwidth
  • 125kHz channel = 0.925MHz radio bandwidth

Organisations

An Organisation can be used to let Organisations or teams manage their own applications and optionally their own gateways.

An Organisation can have:

  • Service-profiles
  • Device-profiles
  • Gateways
  • Applications
  • Users
Service-profiles

Global admin users are able to manage service-profiles for a given Organisations. These service-profiles can then be used by (Organisation) admin users when creating applications.

Device-profiles

Device-profiles can be created by (Organisation) admin users and can be assigned when creating a device.

Gateways

An Organisation can manage its own set of gateways. Note that when an Organisation is created by a global administrator, it can decide that an Organisation can not have any gateways. In this case, the gateway option is not available to an Organisation. That an Organisation is able to manage its own set of gateways does not mean that the coverage is limited to this set of gateways. Gateways connectivity will be shared across the whole network.

Applications

Applications can be created by (Organisation) admin users and define a group of devices with the same goal. For example, following the weather condition or air quality.

Users

Users can be assigned to an Organisation to grant them access to the Organisation. Within the context of that assigment, an user can be an Organisation administrator or a regular user.

Organisation admin

An Organisation administrator is authorized to manage the users assigned with the Organisation and manage the gateways, applications and nodes of the gateway.

Regular user

Regular users are able to see all data, but are not able to make any modifications.

Service-profile Management

The service-profile can be seen as the “contract” between an user and the network. It describes the features that are enabled for the user(s) of the service-profile and the rate of messages that can be sent over the network. When creating a service-profile, LPWAN Server will create the actual profile on the selected network-server, and will keep a reference record so it knows to which Organisation it belongs.

Device-profile Management

A device-profile defines the device capabilities and boot parameters that are needed by the network-server for setting the LoRaWAN radio access service. These information elements shall be provided by the end-device manufacturer.

When creating a device-profile, LPWAN App Server will create the actual profile on the selected network-server, and will keep a reference record so it knows to which Organisation it belongs. Changes made to some of the fields will require the reactivation of the device.

Application Management

An application is a collection of devices with the same purpose / of the same type. Think of a weather station collecting data at different locations for example. When creating an application, you need to select the Service-profile which will be used for the devices created under this application. Note that once a service-profile has been selected, it can’t be changed.

Devices

Multiple devices can be added to each application.

Multicast-groups

By creating a multicast-group, it is possible to send a single downlink payload to a group of devices (the multicast-group). All these devices share the same multicast-address, session-keys and frame-counter.

After creating a multicast-group, it is possible to assign devices to the group. Please note that the device must already created (see devices). Only devices that share the same service-profile as the multicast-group can be added.

Provisioning of the device

The provisioning of the multicast-group on the device happens out-of-band. This means that after adding a device to a multicast-group, you must also configure the device with the multicast-address, session-keys, etc.

Sending data

Sending data to the multicast-group happens using the gRPC or RESTful JSON API.

Device Management

A device is the end-device connecting to, and communicating over the LoRaWAN network. LPWAN Server supports both OTAA (over the air activation) and ABP (activation by personalization) type devices (configured by the selected device-profile).

Create and update

When creating or updating a device, you need to select the device-profile matching the device capabilities. E.g. the device-profile defines if the device is of type OTAA or ABP. In the dropdown, only the device-profiles are available which are provisioned on the same network-server as the service-profile which is assigned to the application above the device.

Tags and variables

Each device can have multiple user-defined tags and variables assigned. Variables are used for integrations and can for example contain API tokens. Tags are exposed when LPWAN App Server published device events and can be used to add additional meta-data, e.g. for aggregation.

Activation
OTAA devices

After creating a device, you can manage the application-key (and network-key for LoRaWAN 1.1 devices) under the Keys (OTAA) tab. Under the Activation you will see the current device activation (if activated).

ABP devices

After creating a device, you can ABP activate this device under the Activation tab. You can either enter the device address, network session key and application session key or generate these. For LoRaWAN 1.1 devices, the network session key is replaced by network session encryption key, serving network session integrity key and forwarding network session integrity key.

Gateway Management

An Organisation is able to manage its own set of gateways. Please note that this feature might be unavailable when the Organisation is configured without gateway support.

That a gateway belongs to a given Organisation does not mean that the usage of a gateway is limited to the Organisation. Every node in the whole network will be able to communicate using the gateway. The Organisation will be responsible however for managing the gateway details (e.g. name, location) and will be able to see its statistics.

Statistics

Gateway statistics are based on the aggregated values sent by the gateway / packet-forwarder. In case no statistics are visible, it could mean that the gateway is incorrectly configured.

Gateway-profiles

When assigning a gateway-profile to a gateway, LPWAN Server will send a configuration update to the gateway when its configuration is out-of-date. Gateway-profiles can be configured by clicking on Network servers in the navbar. Note that when a gateway-profile is assigned to a gateway, you must also configure the LPWAN Gateway Bridge configuration in order to handle configuration updates.

Gateway board configuration

For gateways implementing the v2 reference design which support geolocation capabilities, it is possible to configure one or multiple boards. This allows you to configure the FPGA ID and fine-timestamp AES decryption key per board. When the fine-timestamp AES decryption key is configured, LPWAN Server will automatically decrypt the fine-timestamp once it receives an uplink frame from this gateway.

Frame logging

LPWAN Server makes it possible to log frames sent and received by a gateway or device in realtime. To use this feature, you first need to go to the gateway or device detail page. Once you are on this page, open the Live LoRaWAN frame logs tab.

This is a feature for debugging the communication between the network-server and your devices. Do not use this for integration with your applications. As soon as you open this page, LPWAN Server will subscribe to the frames sent / received by the selected gateway or device. Once a frame is sent or received, it will be displayed without the need to refresh the page.

Gateway frame logs

The frame logs view on the gateway detail page will display all frames sent and received by the selected gateway.

Device frame logs

The frame logs view on the device detail page will display only the frames that could be related to a device. This means that in case you want to debug MIC issues, this view might not show this information and it is better to use the gateway view.

Exposed information

Note that all the displayed data can be expanded by clicking on each key. E.g. > phyPayload: {} 3 keys means you can expand this phyPayload item as it has three sub-items. This makes it possible to inspect (unless encrypted):

  • TX related meta-data (frequency, data-rate, …)
  • RX related-meta-data (RSSI, SNR, timestamps, …)
  • LoRaWAN PHYPayload frames and mac-commands (unless encrypted)
Storing frames

Frames displayed are not stored in a database. As soon as you refresh the page, all data is gone. If you need to store these frames, you can use the download button which will generate a JSON formatted file containing all the frames that are logged since opening the Live frame logs tab.

Event logging

LPWAN App Server makes it possible to log events sent to the MQTT broker or configured integrations. To use this feature, you fist need to go to the device detail page. Once you are on this page, open the Live event logs tab.

This is for debugging purposes only! Do not use this for integration with your applications. As soon as you open this page, LPWAN App Server will subscribe to the events of the selected device. Once an event is received, ti will be displayed without the need to refresh the page.

Exposed events

Note that all the displayed data can be expanded by clicking on each key. E.g. > payload: {} 9 keys means you can expand this payload item as it has nine sub-items.

M2M Wallet

The M2M Wallet's Payment Mechanism cleverly enables devices to autonomously process financial transactions within seconds, providing a state-of-the-art device payment mechanism and laying the foundation for automated payments between wireless devices.

Learn more about M2M Wallet's payment mechanism here.

Requirements

To use MXC Foundation's LPWAN Server it is required a login with a valid email.

Get Started with your M2M Wallet

Welcome to MXProtocol. In only a few steps you are going to be part of a state of the art network machine to machine communication.

  1. To be a part of the network you only need a valid email. So start by clicking Register and submitting your email. In a few minutes, you are going to receive a confirmation email with a link to proceed with your registration. If that email does not arrive in your inbox in a few minutes, please check your Spam folder or consider re-submitting your registration request.
  2. Click or paste the URL into your browser. You will find a complete registration form, where you can add and confirm a password as well as pick a name for your Organisation. Learn more about Organisations here. Please note that your Organisation name is to be added with no spaces between characters.
  3. You are all set. On the left navigation menu, you will find the M2M Wallet access button. Click to proceed.
  4. The first step is adding an ETH Account Address. By synchronizing your personal address to your M2M Wallet you will have easy and secure access to exchanges, so you can intuitively Top up your wallet with MXC to power your data network. Click ETH Address on the left navigation and paste your ETH Address. Add your login and password and click confirm. Next, you should see a new field that appeared, showing your Current ETH Address. You can also check your current and past ETH Addresses by clicking History. Please note that for security reasons, the M2M Wallet allows each Organisation to synchronize one ETH Address at a time. You can edit and change your Address whenever you like.
  5. The next step is to Add MXC into your M2M Wallet. You can do this by using any exchange offering MXC. Check all the exchanges here. Make sure you use the right destination Address, which can be found by clicking Top-up and copying the displayed number. Learn more about Top Up by clicking here.
  6. You can check the status of all of your transactions by accessing the History menu. Please note that each transaction takes a few minutes to be processed on the blockchain. If you find any problems with your transaction, you will find support from our team by sending us a message on support@mxc.org

You are all set, Congratulations! Please check our documentation on Staking, Withdraw and setting up an LPWAN Server to find out everything you can do with your new M2M Wallet.

Staking

Staking is a mechanism for democratization of the data network through the MXProtocol. MXC holders can take part in decisions on the network by stating their position through staking at supernodes. Every user gets a fair vote in the first “Data Senate” and earnings are also distributed accordingly. You can find more about Staking MXC here. You can start your first Stake at any point of the month. You can only have one Stake at a time and only in your supernode of choice, your LPWAN Network provider.

Users are able to harvest the core Staking features: set an amount of MXC on the Supernode and get Staking revenue monthly.

Requirements

Staking is possible for M2M Wallet users with Admin status.

Getting Started
  1. Click Staking on the side navigation menu and click the button Set Stake.
  2. Pick amount of Stake and know your estimated revenue rate. Click confirm.
  3. Congratulations, your Stake is set. You can check the History menu to see your investment performance.
  4. Once every 30 days, revenue is added directly to your M2M Wallet balance. Check the History menu. Proportional revenue is acquired if you Unstake in between this 30 days period. Please note that a stake that is removed before the end of the first 30 days will earn no revenue.
  5. To Unstake, click the staking menu again. Now you can see your active stake and click Unstake to withdraw the full amount. Note that a new stake is only possible if you have no active stake.

Top up

You can use any Wallet to Top up your M2M Wallet, including AXS Wallet.

How to Top up your M2M Wallet
  1. Register the ETH address you plan to send tokens in the Supernode portal. Go to your M2M Wallet, click ‘ETH Address’ and add to your Account. Make sure that the ETH Address on the M2M Wallet is the same as the one you registered on your Wallet of choice.
  2. Add coins to your Wallet from your prefered exchange. Find a list of exchanges offering MXC here.
  3. On a new browser tab, login to your M2M Wallet and click ‘Top up’. Copy the ETH Address you find under ‘To ETH Account’. If your M2M Wallet displays the message ‘Can not find any account.’ under ‘From ETH Address’, please check again the step 1 and make sure you have a current ETH Address added to your M2M Wallet.
  4. Go back to your Wallet. Paste the ETH Address as destination of your coin transfer.
  5. Pick an amount and confirm. That’s it, in a few minutes the transaction will be processed and you will be able to check and use your updated M2M Wallet balance to power data transactions.
